1. Executive Summary & Product Overview

This document outlines the complete technical plan for building an AI-powered Data Analytics SaaS platform. The product enables users to upload datasets in CSV, Excel, and JSON formats, analyze them using AI (powered by Anthropic's Claude API), generate interactive visualizations, and create comprehensive reports through natural language queries.

What the Product Does

Dataset Management

Upload CSV, Excel, and JSON files. The system auto-detects schemas, data types, and generates previews. Datasets are stored securely and linked to user accounts.

AI-Powered Analysis

Chat with your data using natural language. Claude AI analyzes datasets, identifies patterns, suggests insights, and generates statistical summaries automatically.

Smart Visualizations

AI recommends the best chart types for your data. Interactive Recharts-powered visualizations with export capabilities for reports and presentations.

Report Generation

Compile multiple analyses into structured reports. Export as PDF or share via link. Reports include charts, insights, and data summaries.

Target Users

  • Business Analysts — who need quick insights without writing SQL or Python
  • Product Managers — who want to explore metrics and generate reports for stakeholders
  • Small Business Owners — who have spreadsheet data but lack analytical expertise
  • Data Teams — who want AI-assisted exploration as a complement to their existing tools

Platform Architecture Overview

The platform is divided into four main sections, each with its own layout:

  • Marketing Site (marketing) — Public-facing landing page, pricing, and SEO-optimized content
  • Auth Pages (auth) — Login, signup, password reset with centered minimal layout
  • App Dashboard (app) — Protected workspace with sidebar navigation for authenticated users
  • Admin Panel (admin) — Administrative controls for user management, revenue tracking, and platform health

2. Technology Stack & Rationale

Every technology choice is deliberate, optimized for a serverless-first SaaS that can scale from zero to production with minimal operational overhead.

LayerTechnologyRationale
Framework Next.js 15 (App Router) Server-side rendering for marketing SEO, API routes for backend, React Server Components for performance, middleware for auth guards. App Router's route groups enable separate layouts per section.
Language TypeScript Type safety across the full stack — Prisma generates types from DB schema, API routes have typed request/response, components have typed props. Catches bugs at compile time rather than production.
Styling Tailwind CSS 4 + shadcn/ui Tailwind 4 uses CSS-first config (no tailwind.config.ts). shadcn/ui provides accessible, customizable components that are copied into the project (not a dependency), giving full control. Dark mode via CSS variables.
Auth Auth.js v5 (NextAuth) JWT sessions for serverless (no session DB lookups), supports Google OAuth + email/password + magic links. Prisma adapter auto-manages Account/Session tables. Built-in CSRF protection.
Database PostgreSQL (Neon) + Prisma Neon provides serverless Postgres with auto-scaling and a generous free tier. Prisma ORM gives type-safe queries, auto-generated migrations, and a visual studio for DB exploration.
Payments Stripe Industry standard for SaaS billing. Handles subscription lifecycle, payment collection, invoicing, tax calculation, and customer portal. Webhook-driven architecture ensures data consistency.
AI Anthropic Claude API 200K context window handles large datasets. Structured output for chart configs. Superior instruction following for data analysis prompts. Cost-effective for analytical workloads.
Analytics GTM + GA4 + PostHog + Pixels GTM as universal container — add/remove ad pixels without code changes. PostHog for product analytics (session replay, funnels, feature flags). Dual tracking ensures both marketing and product teams have the data they need.
File Storage Vercel Blob Serverless file storage with CDN distribution. No S3 configuration needed. Integrates natively with Next.js API routes.
Charts Recharts React-native charting library. Declarative API makes it easy for AI to generate chart configurations. Supports all common chart types (bar, line, area, pie, scatter).
Deployment Vercel Zero-config deployment from Git. Edge middleware for auth, built-in CDN, automatic HTTPS, preview deployments for every PR.
Why Serverless-First?

Every component (Neon DB, Vercel Functions, Vercel Blob, Stripe) scales to zero when idle and auto-scales under load. This means $0/month cost at zero users and linear cost scaling as the user base grows. No servers to manage, no containers to orchestrate — ideal for a launch-stage SaaS.

Route Groups Explained

Parenthesized folders like (marketing) create route groups in Next.js App Router. They organize routes with shared layouts without affecting the URL path. So (marketing)/pricing/page.tsx maps to /pricing, not /marketing/pricing. This lets each section have its own layout.tsx — marketing gets a navbar+footer, the app gets a sidebar, auth gets a centered card.

4. Database Schema Design

The database uses 7 core models managed by Prisma ORM. The schema is designed for a multi-tenant SaaS with subscription-based access control.

Entity Relationship Overview

User ──┬── Subscription (1:1) ──── Stripe Customer/Subscription ├── Dataset[] (1:many) ──── Report[] (1:many per dataset) ├── Account[] (1:many) ──── OAuth providers (Google, etc.) ├── Session[] (1:many) ──── Active sessions └── AuditLog[] (1:many) ──── Action history

Model Details

User Model

Central entity. Stores identity, role, and links to all user-owned resources.

1:1 with User. Tracks Stripe subscription state and usage limits per billing period.

1.5 Prisma Client Singleton

In development, Next.js hot-reloads create new Prisma instances on each reload, exhausting database connections. The singleton pattern stores the client on globalThis to reuse across reloads: 

// src/lib/prisma.ts
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }
export const prisma = globalForPrisma.prisma || new PrismaClient()
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma

Phase 2: Authentication System

~8 files · Auth.js v5, login, signup, password reset, user menu

2.1 Auth.js v5 Configuration

Auth.js v5 (the successor to NextAuth.js) is configured in src/lib/auth.ts. It exports the auth function (for server-side session checks), handlers (for API routes), and signIn/signOut functions.

Authentication Providers

ProviderUse CaseHow It Works
Google OAuth One-click social login User clicks "Continue with Google" → OAuth flow → Account record created → JWT issued
Credentials Email + password login User submits form → bcrypt compare against stored hash → JWT issued on match
Email (Magic Link) Passwordless login User enters email → verification link sent → click link → auto-login with JWT
JWT vs Database Sessions

We use JWT strategy instead of database sessions because: (1) Serverless functions are stateless — JWT carries all session data in the cookie, (2) No database round-trip on every request, (3) Edge middleware can read JWTs without a database connection. Trade-off: JWTs can't be individually revoked (only by changing the secret, which revokes all).

2.2 Auth Callbacks

Auth.js callbacks customize the JWT and session data flow:

  • jwt callback: Adds user.id and user.role to the JWT token on sign-in
  • session callback: Exposes token.sub (user ID) and token.role on the client-side session object

This means any component calling useSession() or auth() gets the user's role, enabling admin checks without a database query.

Phase 5: API Routes

~10 files · Stripe, AI, datasets, admin endpoints

5.1 API Route Architecture

All API routes are Next.js Route Handlers (serverless functions). They follow a consistent pattern:

export async function POST(request: Request) {
  // 1. Authenticate — get session from Auth.js
  const session = await auth()
  if (!session) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  // 2. Validate — parse request body with Zod
  const body = schema.parse(await request.json())

  // 3. Authorize — check subscription limits
  // 4. Execute — perform business logic
  // 5. Audit — log the action
  // 6. Return — JSON response
}

5.2 Stripe Routes

POST /api/stripe/checkout

Creates a Stripe Checkout Session for plan upgrades. Receives priceId in the body. Creates a Stripe customer if one doesn't exist, then generates a checkout URL. Returns the URL for client-side redirect.

POST /api/stripe/portal

Creates a Stripe Customer Portal session for subscription management. The portal is hosted by Stripe and handles: plan changes, payment method updates, invoice viewing, and cancellation. Returns the portal URL.

POST /api/stripe/webhook

Receives Stripe webhook events. Critical: This route must NOT use auth middleware — it's called by Stripe's servers, not by authenticated users. It verifies the webhook signature using the webhook secret.

EventAction
checkout.session.completedCreate/update Subscription record, set plan to PRO or ENTERPRISE
customer.subscription.updatedSync plan and status changes (upgrades, downgrades)
customer.subscription.deletedRevert user to FREE plan
invoice.payment_succeededReset queriesUsed counter for new billing period
invoice.payment_failedSet subscription status to "past_due", send notification
Webhook Idempotency

Stripe may send the same webhook event multiple times. The webhook handler should be idempotent — processing the same event twice should not create duplicate records or charge users twice. We achieve this by checking if the subscription already matches the event data before updating.

5.3 AI Routes

POST /api/ai/analyze

Accepts datasetId and prompt. Fetches the dataset from storage, sends to Claude with a structured analysis prompt, returns insights + chart configs. Checks subscription query limits before processing. Increments queriesUsed on success.

POST /api/ai/query

Natural language query against a dataset. Claude converts the question into data filtering/aggregation operations, executes them against the dataset, and returns formatted results. Supports follow-up queries with conversation context.

5.4 Dataset Routes

GET /api/datasets

Lists all datasets for the authenticated user. Returns id, name, format, rowCount, columns, createdAt.

POST /api/datasets/upload

Handles multipart file upload. Process: receive file → validate format and size → upload to Vercel Blob → parse file to detect schema → create Dataset record with metadata → return dataset info.

DELETE /api/datasets/[id]

Deletes dataset record and associated file from Vercel Blob. Also cascades to delete related reports.

5.5 Admin Routes

Protected by admin role check. GET /api/admin/users returns paginated user list with subscription info. GET /api/admin/stats returns platform metrics: total users, MRR, active subscriptions by plan, API usage totals.

Phase 6: AI Engine

~3 files · Claude API wrapper, file parser, usage tracking

6.1 Claude API Service (src/lib/ai.ts)

Central AI service that wraps all Claude API interactions. Four main functions:

analyzeDataset(data, prompt)

Sends the dataset schema + sample rows + user prompt to Claude. The system prompt instructs Claude to return structured JSON:

{
  "insights": [
    { "title": "Revenue Trend", "description": "Revenue increased 23% QoQ...", "importance": "high" }
  ],
  "charts": [
    {
      "type": "bar",
      "title": "Revenue by Quarter",
      "data": [{ "quarter": "Q1", "revenue": 45000 }, ...],
      "xKey": "quarter",
      "yKey": "revenue"
    }
  ],
  "summary": "The dataset shows strong growth with seasonal patterns..."
}

This structure maps directly to Recharts components on the frontend.

generateChartConfig(data, prompt)

AI recommends the optimal visualization for given data. Returns chart type (bar, line, pie, scatter, area), axis mappings, color scheme, and aggregation method.

queryData(data, query)

Converts natural language to data operations. Claude analyzes the schema, determines the right filter/group/aggregate operations, and returns results. Example: "Show me average salary by department" → groups by department column, averages salary column.

generateReport(data, analyses)

Takes multiple previous analysis results and compiles them into a cohesive report structure with executive summary, key findings, detailed charts, and recommendations.

Prompt Engineering Strategy

All prompts use Claude's system prompt for role definition ("You are a data analyst..."), include the full dataset schema in the user message, and use structured output formatting instructions. The system prompt specifies the exact JSON structure expected, and Claude's instruction-following capability ensures consistent, parseable responses.

6.2 File Parser (src/lib/upload.ts)

Handles parsing of uploaded files into a standardized internal format:

FormatParserFeatures
CSV papaparse Auto-detects delimiter, handles quoted fields, streaming for large files, header row detection
Excel (.xlsx) xlsx (SheetJS) Multi-sheet support (uses first sheet), date format detection, formula evaluation
JSON Native JSON.parse Supports array of objects and nested structures, auto-flattens one level deep

After parsing, the system performs schema detection:

  • Column name extraction from headers
  • Type inference: string, number, date, boolean (samples first 100 rows)
  • Null/missing value counting per column
  • Preview generation (first 100 rows for quick display)

6.3 Usage Tracking & Rate Limiting

Each subscription tier has defined limits. Before any AI operation:

  1. Fetch user's Subscription record
  2. Check queriesUsed < queriesLimit
  3. If limit reached, return 429 with upgrade prompt
  4. On success, increment queriesUsed atomically
  5. Log to AuditLog for tracking

Phase 7: Analytics & Marketing Pixels

7.1 Analytics Architecture

The analytics system uses a dual-track approach: GTM for marketing attribution and PostHog for product analytics.

7.2 GTM Integration

Google Tag Manager acts as a container for all marketing pixels. Two script tags are added to the root layout:

  • <script> in <head>: GTM JavaScript snippet
  • <noscript> in <body>: Fallback iframe for users with JS disabled

All ad platform pixels (GA4, Meta, LinkedIn, Twitter/X) are configured inside the GTM web console, not in application code. This means adding or removing a pixel requires zero code changes and zero deployments.

7.3 PostHog Integration

PostHog client initializes in a React context provider wrapping the entire app. It provides:

  • Auto-capture: Automatically tracks clicks, page views, and form submissions
  • Session replay: Records user sessions for UX debugging (configurable sampling rate)
  • Feature flags: Server-side and client-side feature flag evaluation
  • A/B testing: Built on feature flags, integrates with analytics for experiment analysis
  • Custom events: Via posthog.capture() for product-specific tracking

7.4 Unified Event System

A single track() function dispatches events to both GTM and PostHog simultaneously:

// src/lib/analytics/events.ts
export function track(event: string, properties?: Record<string, any>) {
  // GTM
  window.dataLayer?.push({ event, ...properties })
  // PostHog
  posthog?.capture(event, properties)
}

15. Data Flow Architecture

File Upload Flow

User drops file | v Frontend: validate format + size against plan limits | v POST /api/datasets/upload (multipart FormData) | v Server: parse file (papaparse/xlsx/JSON.parse) | v Detect schema: columns, types, row count | v Upload raw file to Vercel Blob --> fileUrl | v Create Dataset record in Prisma (metadata + fileUrl) | v Return dataset info to frontend | v Frontend: show in dataset table + success toast

AI Analysis Flow

User types question in chat | v POST /api/ai/analyze { datasetId, prompt } | v Server: auth check --> subscription limit check | v Fetch dataset from Vercel Blob | v Parse dataset, prepare context (schema + sample rows) | v Send to Claude API with structured output prompt | v Claude returns JSON: { insights[], charts[], summary } | v Increment queriesUsed, log to AuditLog | v Return structured response to frontend | v Frontend: render insights as text, charts via Recharts, summary as markdown, "Save to Report" button

Billing Flow

User clicks "Upgrade to Pro" | v POST /api/stripe/checkout { priceId } | v Create Stripe Checkout Session (or Customer if needed) | v Redirect user to Stripe-hosted checkout page | v User completes payment on Stripe | v Stripe sends webhook: checkout.session.completed | v POST /api/stripe/webhook (signature verified) | v Update Subscription: plan=PRO, limits=500 queries | v User returns to /billing, sees updated plan

16. Security Considerations

ThreatMitigation
Unauthorized access Edge middleware auth check on all protected routes; JWT verification on API routes
Privilege escalation Role check at middleware + API level (defense in depth); role stored in JWT, verified server-side
Data leakage All dataset queries filter by userId — users can only access their own data
File upload attacks Allowlist of file extensions (csv, xlsx, json); file size limits per plan; content-type validation
Stripe webhook forgery Verify webhook signature using STRIPE_WEBHOOK_SECRET before processing
AI prompt injection Dataset content is treated as data (user message), not instructions (system prompt); output is JSON-parsed, not executed
XSS React's default escaping; AI output rendered as markdown (sanitized), not raw HTML
CSRF Auth.js built-in CSRF token validation on all auth actions
SQL injection Prisma ORM uses parameterized queries — no raw SQL
Brute force Rate limiting on auth endpoints; account lockout after repeated failures
Data at rest Neon encrypts data at rest (AES-256); passwords bcrypt-hashed with salt
Data in transit HTTPS enforced by Vercel; all API calls use TLS
